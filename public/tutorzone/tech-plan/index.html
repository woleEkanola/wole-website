Technical Implementation Plan

Complete technical architecture, database schema, API design, and implementation strategy for the Tutorszone platform. This document assumes familiarity with the MVP features document and the full PRD.

High-Level Architecture

The platform follows a monolithic Next.js full-stack pattern with a service layer pattern internally. This keeps the MVP lean — one codebase, one deploy, one database — while the service abstraction layer makes it possible to split into microservices later if needed.

┌────────────────────────────────────────────────────────────┐
│                    CLIENT (Browser)                        │
│        Next.js App Router (SSR + Client Components)        │
│                 Role-based routing & layouts                │
├────────────────────────────────────────────────────────────┤
│                    API LAYER (Next.js Route Handlers)       │
│  /api/auth/*  │ /api/schedule/* │ /api/classroom/* │ ...   │
├────────────────────────────────────────────────────────────┤
│                    SERVICE LAYER (lib/services/)            │
│  AuthService │ ScheduleService │ ClassroomService │ ...    │
├────────────────────────────────────────────────────────────┤
│  Prisma ORM (type-safe DB access + multi-tenant middleware) │
├────────────────────────────────────────────────────────────┤
│  PostgreSQL 18  │  Redis (BullMQ)  │  Local Disk Storage   │
└────────────────────────────────────────────────────────────┘
        │                │
   ┌────▼────┐    ┌──────▼──────┐
   │ LiveKit  │    │   Twilio    │
   │ (SFU)    │    │ SMS/WhatsApp│
   └─────────┘    │ Masked Call │
                  └─────────────┘

Key architectural decisions:

  • Monolith-first: Single Next.js app handles frontend and API. Service layer abstracts business logic so it can be extracted later.
  • Prisma middleware pattern: Every Prisma query is automatically scoped to the current tenant via a tenant_id filter injected by middleware.
  • Event-driven background jobs: BullMQ (Redis-backed) handles billing, notifications, screenshot capture, report generation, and scheduled cleanup.
  • LiveKit abstraction: A thin adapter layer makes switching between self-hosted and LiveKit Cloud a config change.

Core Database Schema

Every table includes tenant_id from day one. This is the single most important architectural constraint — it makes the V1→V2 multi-tenant migration a configuration change rather than a data migration.

Multi-Tenant Strategy

V1 (Single-Tenant): Hardcode tenant_id to a single value. All tables have the column, but it's always the same value. Prisma middleware injects the filter.

V2 (Multi-Tenant): Determine tenant_id from the request context (subdomain, JWT claim, or header). Same middleware — just the source of the tenant ID changes.

Implementation:

// lib/prisma.ts — tenant-aware Prisma client wrapper
import { PrismaClient } from '@prisma/client';

const globalPrisma = new PrismaClient();

// Middleware injects tenant_id on every query
globalPrisma.$use(async (params, next) => {
  const tenantId = getTenantId(); // From JWT or hardcoded in V1

  // Skip for tenant-agnostic tables
  if (['tenants', 'audit_logs'].includes(params.model)) return next(params);

  // Add tenant filter to all queries
  params.args.where = { ...params.args.where, tenant_id: tenantId };
  return next(params);
});

export default globalPrisma;

JWT Claims: { sub: user.id, email, role, tenant_id, exp, iat }

Middleware (middleware.ts):

  • Public routes: /, /login, /register — no auth required
  • Role routes: /dashboard/:role/* — checks JWT role matches path role
  • API routes: /api/* — validates JWT, injects user context
  • LiveKit tokens: Short-lived JWT issued by API to authorize room join

Key System: LiveKit Integration

Abstraction Layer

// lib/livekit.ts
import { AccessToken, RoomServiceClient } from 'livekit-server-sdk';

const LIVEKIT_CONFIG = {
  url: process.env.LIVEKIT_URL,
  apiKey: process.env.LIVEKIT_API_KEY,
  apiSecret: process.env.LIVEKIT_API_SECRET,
};

const roomClient = new RoomServiceClient(
  LIVEKIT_CONFIG.url, LIVEKIT_CONFIG.apiKey, LIVEKIT_CONFIG.apiSecret
);

export function createRoomToken(roomName: string, userId: string, isObserver = false) {
  const token = new AccessToken(LIVEKIT_CONFIG.apiKey, LIVEKIT_CONFIG.apiSecret, {
    identity: userId, ttl: '4h',
  });
  token.addGrant({ roomJoin: true, room: roomName, canPublish: !isObserver, canSubscribe: true });
  return token.toJwt();
}

export { roomClient };

Silent Observer Flow

  1. LES clicks "Observe" on a live session from QA dashboard
  2. API calls createRoomToken(roomName, lesUserId, isObserver=true)
  3. Token grants canPublish: false — LES can see/hear but not transmit
  4. LES joins the room invisibly — no participant notification

Attendance Tracking

  1. LiveKit webhook fires on participant_joined and participant_left
  2. Webhook handler at /api/webhooks/livekit validates signature
  3. Inserts row into attendance_logs with timestamps
  4. If join time > 5 min after session start → is_late = true
  5. If no join within 15 min → BullMQ job marks session as no_show

Key System: Scheduling Engine

Timezone Handling

  • All times stored in UTC in the database
  • User's timezone detected from Intl.DateTimeFormat().resolvedOptions().timeZone
  • Conversion happens at the presentation layer: dayjs.utc(dbTime).tz(userTimezone)
  • Scheduling UI shows both scheduler's and student's timezone side by side

Conflict Detection Algorithm

// lib/services/schedule.service.ts
async function hasConflict(tutorId: string, startTime: Date, endTime: Date, excludeId?: string) {
  const overlapping = await prisma.sessions.findFirst({
    where: {
      tutor_id: tutorId,
      status: { in: ['scheduled', 'live'] },
      id: { not: excludeId },
      AND: [
        { start_time: { lt: endTime } },
        { end_time: { gt: startTime } }
      ]
    }
  });
  return !!overlapping;
}

Recurring Sessions

  • Store recurrence_rule as RFC 5545 RRULE string (e.g., FREQ=WEEKLY;BYDAY=MO,WE;COUNT=12)
  • Generate concrete session instances for the next 4 weeks
  • Cron job generates new instances weekly for the next rolling window

Key System: Notification Pipeline

Multi-Channel Delivery

// lib/services/notification.service.ts
async function sendNotification(userId: string, template: string, data: Record) {
  const user = await prisma.users.findUnique({ where: { id: userId } });
  const prefs = await getUserPreferences(userId);

  const jobs = [];
  if (prefs.email_opt_in)  jobs.push(emailQueue.add('send', { to: user.email, template, data }));
  if (prefs.sms_opt_in)    jobs.push(smsQueue.add('send', { to: user.phone, template, data }));
  if (prefs.whatsapp_opt_in) jobs.push(whatsappQueue.add('send', { to: user.phone, template, data }));

  await Promise.all(jobs);
}

Key System: Reports

Report Lifecycle

  1. Draft: Tutor creates report after assessment session. Form pre-fills student info, subject, and topics from session data.
  2. Submitted: Tutor submits for review. LES sees report in pending queue.
  3. Approved: LES reviews and approves. Report is locked from editing. BullMQ job triggers auto-send to parent via SendGrid. Parent can also access from dashboard.
  4. Rejected: LES returns report to tutor with feedback for revision.

import puppeteer from 'puppeteer';

async function generateReportPDF(report: Report) {
  const browser = await puppeteer.launch({ headless: true });
  const page = await browser.newPage();
